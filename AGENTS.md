# AGENTS.md — Rules every AI agent must follow

This file is the **contract** for any AI agent (or human) that touches this
repository. **Read it before making changes.** It is intentionally short,
prescriptive, and non-negotiable. If a rule blocks you, escalate (see
[§5](#5-multi-agent-delegation-model)) — do not work around it.

> Referenced from [`README.md`](README.md). The rules below are also tracked as
> user stories in [`docs/backlog/backlog.json`](docs/backlog/backlog.json)
> (epic **E22**).

---

## 1. Prime directives

1. **Story-first.** No feature is built without a **user story** first. Add it to
   [`docs/backlog/backlog.json`](docs/backlog/backlog.json), seed it as a GitHub
   issue, **then** implement. Every change links the issue it implements.
2. **Never deny, never drop.** An agent either **completes** a task or
   **explicitly delegates** it to a named role. No task is silently abandoned.
3. **Isolation is sacred.** Multi-tenant data isolation (RLS) is never weakened.
   Every new tenant-scoped table ships with an RLS policy in the same change.
4. **Prove it.** A change is not done until it is **validated** (parse/lint/test)
   and the original goal is demonstrably met.

---

## 2. The feature workflow (story-first)

```
idea / request
  → write a user story in backlog.json  (As a <role>, I want <goal>, so that <benefit> + acceptance criteria)
  → seed it as a GitHub issue           (scripts/github/seed-backlog.ps1, idempotent by title)
  → implement against the acceptance criteria
  → validate (pglast / lint / build / test)
  → commit referencing the issue (e.g. "… (closes #N)")
```

- **User stories** use the canonical form: _As a `<role>`, I want `<goal>`, so
  that `<benefit>`_, each with a checklist of **acceptance criteria**.
- Use the existing **roles** vocabulary (super-admin, district admin, school
  admin, instructional designer, teacher, TA, student, parent, compliance
  officer, platform engineer).
- The seeder is **idempotent by issue title** — edit `backlog.json` and re-run;
  it will not duplicate existing issues.

## 3. Technical guardrails (hard rules)

| Area | Rule |
| ---- | ---- |
| **RLS** | Any new tenant-scoped table is added to the `tenant_tables` list in [`database/policies/rls.sql`](database/policies/rls.sql). Tables without their own `tenant_id` (e.g. `role_permission`) get a join-based isolation policy. |
| **Schema** | `database/schema.sql` is the source of truth. Validate every edit with **pglast** (`python -c "import pglast; pglast.parse_sql(open('database/schema.sql',encoding='utf-8').read())"`) before committing. |
| **Control plane** | The `tenant` table is control-plane and is **not** in the RLS `tenant_tables` loop — do not add it. |
| **Generated docs** | Per-service specs in `docs/services/` are generated by `scripts/docs/gen-service-specs.py`. Edit the **script**, then regenerate — never hand-edit the output. |
| **PowerShell scripts** | Keep `.ps1` files **ASCII-only** (Windows PowerShell 5.1 reads `-File` scripts as ANSI without a BOM and breaks on non-ASCII). Read JSON with `Get-Content -Raw -Encoding UTF8`. |
| **JSON files** | Write without a BOM (use Python/Node or `[System.IO.File]::WriteAllText` with `UTF8Encoding($false)`) — a BOM breaks strict JSON parsers. |
| **Identity / secrets** | Never commit secrets. Silo DSNs are resolved from `tenant.database_ref` via the secret store, never hard-coded. Credentials are delegated to the external CIAM — no home-grown IdP. |
| **Events** | State changes write a transactional **outbox** row in the same DB transaction; consumers dedupe via **inbox** + `idempotency_key`. |
| **Commits** | **Never** add a `Co-authored-by: Copilot` trailer. Use Conventional Commit prefixes (`feat`, `fix`, `docs`, `chore`, …) and reference the issue. |

## 4. Definition of done

A change is done only when **all** of these hold:

- [ ] A user story / issue exists and is linked.
- [ ] Acceptance criteria are met.
- [ ] New tenant tables have RLS; `schema.sql` validates with pglast.
- [ ] Relevant build / lint / test pass (run the repo's existing tooling only).
- [ ] Generated artifacts (service specs) are regenerated, not hand-edited.
- [ ] Commit references the issue; working tree is clean and pushed.

## 5. Multi-agent delegation model

Work is decomposed across **specialised agent roles**. A fresh agent may be spun
up per task, but every agent operates under one of these roles and follows the
**hand-off protocol**. **No agent denies a task** — it either does it or delegates
it to the role that can.

### Roles

| Role | Owns | Hands off to |
| ---- | ---- | ------------ |
| **Orchestrator** | Decomposes a request into tasks, assigns roles, tracks completion. | any role |
| **Backlog agent** | Writes user stories + acceptance criteria; seeds issues. | Orchestrator |
| **Schema agent** | `database/schema.sql`, RLS policies, migrations; pglast validation. | Service agent, Review agent |
| **Service agent** | A bounded context under `services/*` (one owner per service) + its spec. | Schema agent (for tables), Review agent |
| **Review agent** | Verifies acceptance criteria, isolation, and DoD; requests changes. | the implementing role |
| **Docs agent** | README, `docs/*`, generated specs. | Review agent |

### Hand-off protocol

1. **Accept or delegate — never deny.** If a task is outside your role, **hand it
   to the named role that owns it** (do not refuse and stop). Record the hand-off
   as an issue comment or commit reference naming the receiving role.
2. **Complete context on hand-off.** Each agent is stateless; the delegating agent
   passes everything the receiver needs (issue link, files, acceptance criteria,
   constraints from this file).
3. **Single owner per scope.** Once a scope is delegated, that role owns it until
   done or re-delegated — no duplicated work.
4. **Close the loop.** The Review agent confirms the Definition of Done before a
   task is considered complete; it hands unmet items back to the implementer.

### Escalation (when no role can accept)

If a task genuinely fits no role, or a guardrail blocks all paths, **escalate to
the Orchestrator with a written reason and options** — rather than dropping it or
bypassing a rule. The Orchestrator either creates a new role/story or asks the
human maintainer. The chain always terminates in a decision, never in silence.
